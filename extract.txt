1. WHAT THIS REPO ACTUALLY IS
pixel-agents is a VS Code extension that renders a live pixel art office inside a VSCode webview panel. Each Claude AI agent session running in a terminal is represented as a pixel character walking around a top-down office environment. The characters react in real time to what their AI agent is doing — typing when working, wandering when idle, showing speech bubbles when waiting for permission.
The rendering engine is a pure HTML5 Canvas game loop — no Phaser, no game engine library. It is written entirely in vanilla TypeScript using requestAnimationFrame, a custom sprite renderer, BFS pathfinding, and a finite-state machine for character behaviour.
KEY INSIGHT FOR THE TRADING FIRM: This repo proves the entire pixel office engine can be built without Phaser. The approach is lighter, faster to understand, and gives complete control over every pixel. The trading firm frontend should follow the same canvas-first, no-game-engine pattern.

2.2 The Two-Layer Pattern
The most important architectural pattern in this repo is the clean separation between the canvas layer and the HTML overlay layer. These two layers are always kept separate and never mixed:
CANVAS LAYER — The Phaser/game canvas renders all pixel art: floor tiles, furniture, character sprites, speech bubbles, selection outlines. No HTML inside the canvas.
HTML OVERLAY LAYER — React components positioned absolutely over the canvas render all text labels, tool status badges, status dots, and close buttons. These are standard DOM elements with CSS, positioned using the character's world coordinates converted to screen coordinates each frame via requestAnimationFrame.
This is the critical pattern to copy. The ToolOverlay component runs its own RAF loop, recalculates every character's screen position each frame from world coordinates, and repositions the HTML elements accordingly. Characters move smoothly on canvas; labels follow in DOM. This is cleaner than trying to render text on canvas.
3. THE GAME LOOP
gameLoop.ts implements the standard game loop pattern using requestAnimationFrame. It is minimal and clean — 38 lines total.
// Pattern: startGameLoop(canvas, { update, render }) → cleanup fn
const frame = (time: number) => {
  const dt = Math.min((time - lastTime) / 1000, MAX_DELTA_TIME_SEC)  // capped at 0.1s
  lastTime = time
  callbacks.update(dt)      // advance all character state machines
  ctx.imageSmoothingEnabled = false  // CRITICAL: keep pixel art sharp
  callbacks.render(ctx)     // draw everything
  rafId = requestAnimationFrame(frame)
}
// Returns a cleanup function: () => { stopped = true; cancelAnimationFrame(rafId) }
dt is always in SECONDS not milliseconds — all speed constants use px/sec and sec units throughout
MAX_DELTA_TIME_SEC = 0.1 — caps the delta time to prevent huge jumps after tab switch or debugger pause
imageSmoothingEnabled = false is set on EVERY frame, not just once — browsers sometimes reset it
The loop returns a cleanup function called in useEffect return for proper React unmount handling

TYPE → IDLE:  when isActive becomes false AND seatTimer expires
IDLE → WALK:  when wanderTimer expires, picks random walkable tile and pathfinds
IDLE → WALK:  when isActive becomes true, pathfinds to assigned seat

WALK → TYPE:  when path complete AND arrived at seat while inactive (rest before wandering)
// Key flags on Character object:

ch.seatId        // assigned desk seat uid, or null
ch.path          // remaining BFS path steps as {col, row}[]
ch.wanderTimer   // countdown until next wander decision
ch.wanderCount   // moves made in current roaming cycle
ch.wanderLimit   // max moves before returning to seat for rest
ch.seatTimer     // time to rest at seat before wandering again (2-4 minutes)
4.3 The Wander System
When a character goes inactive (agent stops working) it does not immediately stand up and wander. It follows this sequence:
On setAgentActive(false): seatTimer is set to -1 as a sentinel (meaning 'just ended turn')
While seatTimer > 0: stays seated at desk even though inactive
When seatTimer expires: transitions to IDLE, sets wanderTimer to 2-20 seconds
Each time wanderTimer expires: pathfinds to random walkable tile, increments wanderCount
When wanderCount >= wanderLimit (3-6 moves): pathfinds back to own seat, rests for 2-4 minutes
This creates realistic office behaviour — agents work, occasionally wander, return to desk, repeat
4.4 Re-pathing on State Change
When an agent becomes active mid-walk (wander interrupted by new work), the WALK case checks:
// In WALK state update, every frame:
if (ch.isActive && ch.seatId) {
  const seat = seats.get(ch.seatId)
  const lastStep = ch.path[ch.path.length - 1]
  if (!lastStep || lastStep.col !== seat.seatCol || lastStep.row !== seat.seatRow) {
    // Currently walking somewhere else — repath to seat immediately
    const newPath = findPath(ch.tileCol, ch.tileRow, seat.seatCol, seat.seatRow, ...)
    if (newPath.length > 0) {
      ch.path = newPath  // replace current path mid-walk
      ch.moveProgress = 0
    }
  }
}
// Result: agent smoothly redirects to desk the moment work starts
5. MOVEMENT & PATHFINDING
5.1 Tile-Based Movement with Pixel Interpolation
Characters move tile-by-tile but interpolate smoothly between tiles. The movement is not purely tile-snapping — it produces fluid pixel motion.
// moveProgress is 0→1 lerp between current tile and next tile
ch.moveProgress += (WALK_SPEED_PX_PER_SEC / TILE_SIZE) * dt
// WALK_SPEED_PX_PER_SEC = 48, TILE_SIZE = 16 → crosses one tile in 0.333 seconds
const fromCenter = tileCenter(ch.tileCol, ch.tileRow)   // {x, y} in pixels
const toCenter   = tileCenter(nextTile.col, nextTile.row)
const t = Math.min(ch.moveProgress, 1)
ch.x = fromCenter.x + (toCenter.x - fromCenter.x) * t  // linear interpolation
ch.y = fromCenter.y + (toCenter.y - fromCenter.y) * t
if (ch.moveProgress >= 1) {
  // Snap to tile center, advance path
  ch.tileCol = nextTile.col
  ch.tileRow = nextTile.row
  ch.path.shift()          // remove completed step
  ch.moveProgress = 0
}
5.2 BFS Pathfinding
findPath() in tileMap.ts implements standard BFS on a 4-connected grid (no diagonals). Key implementation details:
Blocked tiles are stored as a Set<string> of 'col,row' keys — O(1) lookup
Own seat tile is temporarily removed from blockedTiles before pathfinding so character can reach it
End tile must be walkable — if a seat tile is blocked by furniture, no path is returned
Returns path EXCLUDING start tile, INCLUDING end tile
Empty path = already at destination or no path exists
// Blocked tiles are furniture footprints registered at layout build time
// Pattern: 'col,row' string keys in a Set
const blockedTiles = new Set<string>(['3,2', '4,2', '3,3'])  // desk footprint example
// Temporarily unblock own seat (character should be able to sit in it):
if (key) this.blockedTiles.delete(key)  // remove own seat
const path = findPath(...)               // pathfind
if (key) this.blockedTiles.add(key)      // restore

6.1 Z-Sorting
Everything in the scene — furniture, wall sprites, and characters — is z-sorted together into a single drawables array. This is how characters appear behind desks when walking behind them and in front when walking in front.
// Z-sort pattern: collect all drawables with a zY value, then sort
interface ZDrawable {
  zY: number          // the Y world-coordinate used for depth sorting
  draw: (ctx) => void // the actual draw call as a closure
}
// Character zY = bottom of character tile + small offset:
const charZY = ch.y + TILE_SIZE / 2 + CHARACTER_Z_SORT_OFFSET  // +0.5
// Furniture zY is set at layout build time (typically bottom edge of footprint)
drawables.sort((a, b) => a.zY - b.zY)  // lower Y = further back = drawn first
for (const d of drawables) d.draw(ctx)  // draw in order

7. THE MATRIX SPAWN/DESPAWN EFFECT
When agents are added or removed from the office, their characters appear/disappear with a Matrix digital rain effect. This is the most visually distinctive animation in the codebase.
7.1 How It Works
On spawn: a green 'rain' sweep moves down the sprite from top to bottom, revealing the character pixel by pixel
On despawn: the same sweep moves down, consuming and erasing pixels behind it
Duration: 0.3 seconds total (MATRIX_EFFECT_DURATION_SEC)
Each column (16 total) has a random stagger offset (0-0.3 of total duration) so columns don't all start simultaneously
A hash-based flicker function creates a shimmer — ~70% of trail pixels are visible at any given frame
7.2 Effect State on Character
// Added to Character object:
ch.matrixEffect:       'spawn' | 'despawn' | null
ch.matrixEffectTimer:  number    // counts UP from 0 to MATRIX_EFFECT_DURATION
ch.matrixEffectSeeds:  number[]  // 16 random values [0-1] for column stagger
// In officeState.update():
if (ch.matrixEffect) {
  ch.matrixEffectTimer += dt
  if (ch.matrixEffectTimer >= MATRIX_EFFECT_DURATION) {
    if (ch.matrixEffect === 'spawn') {
      ch.matrixEffect = null  // clear effect, resume normal FSM
    } else {
      toDelete.push(ch.id)   // despawn complete — remove character
    }
  }
  continue  // SKIP normal FSM while effect is active
}
7.3 Render Logic
// Per-pixel rendering during effect (replaces normal drawImage):
const progress = ch.matrixEffectTimer / MATRIX_EFFECT_DURATION  // 0→1
const totalSweep = ROWS + TRAIL_LENGTH  // 24 + 6 = 30
for each col (0-15):
  stagger = seeds[col] * 0.3  // 0-0.3 delay
  colProgress = (progress - stagger) / 0.7  // normalized 0→1
  headRow = colProgress * totalSweep  // current sweep position
  for each row (0-23):
    distFromHead = headRow - row
    if distFromHead < 0:  // above head
      spawn: invisible    despawn: draw original pixel
    if distFromHead < 1:  // head
      draw bright white-green '#ccffcc'
    if distFromHead < TRAIL_LENGTH:  // trail
      spawn: draw pixel + fading green overlay
      despawn: draw fading green only (pixel consumed)
    else:  // below trail
      spawn: draw original pixel (fully revealed)
      despawn: nothing (consumed)

9. THE HTML OVERLAY LAYER — ToolOverlay
ToolOverlay is a React component that renders pure HTML <div> elements positioned over the canvas. It runs its own requestAnimationFrame loop to keep position in sync with character movement — this is the key technique for combining canvas with DOM overlays.
// ToolOverlay runs its own RAF loop — NOT tied to the game loop:
useEffect(() => {
  let rafId = 0
  const tick = () => {
    setTick(n => n + 1)  // force React re-render every frame
    rafId = requestAnimationFrame(tick)
  }
  rafId = requestAnimationFrame(tick)
  return () => cancelAnimationFrame(rafId)
}, [])
// Each frame, convert world coords to screen coords:
const dpr = window.devicePixelRatio || 1

const screenY = (deviceOffsetY + (ch.y + sittingOffset - TOOL_OVERLAY_VERTICAL_OFFSET) * zoom) / dpr

9.1 Activity Text Logic
The label shown above each character is derived from the agent's tool state using getActivityText():
function getActivityText(agentId, agentTools, isActive): string {
  const tools = agentTools[agentId]
  if (tools && tools.length > 0) {
    // Latest non-done tool:
    const activeTool = [...tools].reverse().find(t => !t.done)
    if (activeTool) {
      if (activeTool.permissionWait) return 'Needs approval'
      return activeTool.status  // e.g. 'Read file: config.py'
    }
    // All tools done but agent still active (between tool calls):
    if (isActive) return tools[tools.length - 1].status  // show last
  }
  return 'Idle'
}

14. CAMERA SYSTEM
The camera is implemented as a panX/panY offset plus a zoom level. The map is always centered in the viewport, with pan applied on top. Camera follow smoothly lerps toward the selected character.
// Center map in viewport:
const mapW = layout.cols * TILE_SIZE * zoom
const offsetX = Math.floor((canvasWidth - mapW) / 2) + Math.round(panRef.current.x)
// Camera follow (runs every render frame in OfficeCanvas):
if (officeState.cameraFollowId !== null) {
  const followCh = officeState.characters.get(officeState.cameraFollowId)
  const targetX = mapW / 2 - followCh.x * zoom  // pan that centers the character
  const dx = targetX - panRef.current.x
  if (Math.abs(dx) < CAMERA_FOLLOW_SNAP_THRESHOLD) {  // 0.5px
    panRef.current = { x: targetX, y: targetY }  // snap when close
  } else {
    panRef.current.x += dx * CAMERA_FOLLOW_LERP  // lerp factor: 0.1
  }
}

15.4 Key Implementation Notes
Keep OfficeState as a plain class outside React — do not put it in Zustand. Zustand is for React-readable state (inbox, trades, risk). The canvas game state stays imperative.
The PhaserBridge pattern from Doc 3 is equivalent to: bridge.emit(event, payload) → registered scene listeners. In this no-Phaser approach, just call officeState methods directly from the WebSocket event router.
The ToolOverlay RAF pattern can be used for the data panel overlays — keep them as HTML positioned over the canvas, not drawn on canvas.
Floor 1, 2, 3 are three separate tile maps — three OfficeLayout objects. Switch between them using the FloorSelector which calls a different Phaser scene start. In this engine: stop the current game loop, load the next floor's layout into OfficeState, restart loop.
The auto-on electronics pattern (monitors lighting up when agent is active) maps directly to agent desk monitors changing state when a trading signal arrives.
